# Deployment Database Migration

This document defines the path from the current single-replica SQLite deployment to managed Postgres. The backend now supports both SQLite and Postgres through the shared repository layer, but production cutover still requires a managed database, secret wiring, data migration, and a canary.

## Current State

The backend supports two database modes:

- Local/default mode uses `sqlite:///` and `infra/k8s/configmap.yaml` sets `DATABASE_URL=sqlite:////app/data/deskclaw.db`.
- Production database mode uses `postgres://` or `postgresql://` through `desk-ai-secrets`, not through the ConfigMap.
- `backend/app/db/session.py` owns SQLite/Postgres connection setup and idempotent schema creation.
- `backend/app/db/decision_log.py`, `backend/app/db/audit.py`, and feedback persistence use the shared `Database` abstraction.
- `infra/k8s/backend.yaml` still runs one backend replica with a `ReadWriteOnce` SQLite PVC by default.
- `scripts/render-release-k8s.sh` can render `DATABASE_MODE=postgres`, remove the backend SQLite PVC wiring, read `DATABASE_URL` from a Secret, and set `BACKEND_REPLICAS`.

SQLite remains acceptable for local development, demos, and a single-replica private pilot. Backend horizontal scaling requires `DATABASE_MODE=postgres`, Postgres integration tests, a successful canary, and a provider-managed backup/PITR plan.

## Data Inventory

Current persisted tables:

| Table | Owner | Purpose | Migration concern |
| --- | --- | --- | --- |
| `decisions` | `DecisionLogService` | Simple feedback events from `POST /api/feedback`. | Text timestamps and nullable recommendation IDs. |
| `app_users` | `AuditRepository` | Actor identity records for trusted actor attribution. | Primary key is actor-provided `actor_id`; preserve exactly. |
| `ai_audit_log` | `AuditRepository` | Redacted AI request/response audit rows and telemetry source data. | JSON is stored as text today; Postgres should use `jsonb` for payload/tool-call columns. |
| `decision_log` | `DecisionLogRepository` | Full workflow decision records from `POST /api/decisions`. | Integer autoincrement IDs must map to a Postgres identity or bigint sequence. |

## Target State

The production target is:

- managed Postgres from the hyperscaler or a managed database provider;
- `DATABASE_URL` delivered through a Secret or external secret controller, not a ConfigMap;
- no backend data PVC for application database state;
- schema creation/migrations owned by the backend deployment process;
- backend repository code using the shared database abstraction for SQLite and Postgres;
- backend horizontal scaling enabled only after Postgres-backed tests and a successful canary.

## Implemented Repo Support

- `psycopg` is installed with the backend package.
- `DATABASE_URL=postgres://...` is normalized to `postgresql://...`.
- Postgres schemas are created for `decisions`, `app_users`, `ai_audit_log`, and `decision_log`.
- AI audit payloads and workflow decision payloads use Postgres `jsonb` while SQLite keeps local JSON text.
- `tests/integration/test_postgres_database.py` exercises the persisted tables against a live Postgres service.
- CI runs a separate `Backend Postgres integration` job.
- `scripts/check-database-runtime.sh` verifies backend health, Secret contents, and Deployment wiring after cutover.
- Release rendering supports:

  ```bash
  DATABASE_MODE=postgres \
    DATABASE_SECRET_NAME=desk-ai-secrets \
    DATABASE_URL_SECRET_KEY=DATABASE_URL \
    BACKEND_REPLICAS=2 \
    ./scripts/render-release-k8s.sh git-<sha> /tmp/desk-ai-postgres-release.yaml
  ```

## Cutover Plan

1. Provision managed Postgres.

   Provision the database outside this repo through the chosen hyperscaler. Required settings:

   - private network access from backend pods;
   - TLS required;
   - encrypted storage;
   - automated backups and point-in-time recovery;
   - least-privilege application user;
   - connection limit sized for backend replica count;
   - maintenance window and upgrade policy.

2. Move `DATABASE_URL` to secrets.

   Store the managed Postgres URL in the provider secret manager or External Secrets and expose it through `desk-ai-secrets` as `DATABASE_URL`. Do not commit the URL and do not place credentials in `infra/k8s/configmap.yaml`.

   External Secret rendering can include the database URL:

   ```bash
   INCLUDE_DATABASE_URL=true \
     SECRET_STORE_NAME=desk-ai-runtime-secrets \
     REMOTE_SECRET_KEY=desk-ai/production/runtime \
     ./scripts/render-external-secret.sh /tmp/desk-ai-external-secret.yaml
   ```

3. Confirm schema shape.

   Current Postgres shape:

   | SQLite field | Postgres target |
   | --- | --- |
   | ISO timestamp text | `timestamptz` |
   | JSON text payloads | `jsonb` |
   | `decision_log.id INTEGER PRIMARY KEY AUTOINCREMENT` | `bigint generated by default as identity` |
   | UUID strings | text, preserving externally supplied actor IDs exactly |

4. Export SQLite data.

   Before cutover, freeze public writes by taking the ingress out of rotation, enabling maintenance mode, or otherwise blocking user traffic. Keep one backend pod running long enough to copy the mounted SQLite file:

   ```bash
   export BACKEND_POD="$(kubectl -n desk-ai get pod -l app=backend -o jsonpath='{.items[0].metadata.name}')"
   kubectl -n desk-ai cp "${BACKEND_POD}:/app/data/deskclaw.db" ./deskclaw.db
   ```

   After the file is copied, scale backend to zero so no new SQLite writes happen during import:

   ```bash
   kubectl -n desk-ai scale deployment/backend --replicas=0
   ```

   Export each table with a deterministic tool or migration script. Avoid ad hoc hand edits of audit payloads.

5. Import and verify Postgres data.

   Import into staging first, then production. Verify:

   - row counts by table;
   - newest and oldest timestamps;
   - representative AI audit payloads still parse as JSON;
   - telemetry dashboard queries match expected aggregates;
   - full workflow decision list returns newest entries first.

6. Canary with one backend replica.

   Deploy the Postgres-enabled backend with one replica first. Run:

   ```bash
   DATABASE_MODE=postgres ./scripts/check-database-runtime.sh https://desk-ai.example.com
   kubectl -n desk-ai rollout status deployment/backend --timeout=600s
   ./scripts/smoke-deploy.sh https://desk-ai.example.com
   ```

   Then run at least one model-backed workflow and inspect `ai_audit_log` and telemetry.

7. Scale backend only after canary success.

   Increase backend replicas only after:

   - Postgres integration tests pass;
   - the canary writes and reads all four persisted tables;
   - admin telemetry works from Postgres;
   - connection pool limits are set;
   - rollback to the previous `git-<sha>` is understood.

## Guardrails

- Do not set `BACKEND_REPLICAS` above one unless the release is rendered with `DATABASE_MODE=postgres`.
- Do not scale backend above one replica while `DATABASE_URL` points at SQLite.
- Do not use a shared SQLite PVC with multiple backend replicas.
- Do not migrate raw unredacted AI payloads; current stored audit payloads should already be redacted.
- Do not remove the SQLite PVC until a backup is retained and the Postgres canary has passed.
- Do not expose the managed database publicly; backend pods should reach it over private networking.

## Rollback

If the Postgres canary fails before writes are accepted, roll back to the last known good SQLite-backed release and restore one backend replica:

```bash
./scripts/render-release-k8s.sh git-<last-sqlite-sha> /tmp/desk-ai-sqlite-rollback.yaml
kubectl apply -f /tmp/desk-ai-sqlite-rollback.yaml
kubectl -n desk-ai scale deployment/backend --replicas=1
kubectl -n desk-ai rollout status deployment/backend --timeout=600s
./scripts/smoke-deploy.sh https://desk-ai.example.com
```

If the Postgres canary accepted writes, treat rollback as a data migration problem, not just an image rollback. Preserve Postgres, export new rows, and decide whether to replay them into SQLite or keep the app in maintenance mode until the Postgres fix is deployed.
